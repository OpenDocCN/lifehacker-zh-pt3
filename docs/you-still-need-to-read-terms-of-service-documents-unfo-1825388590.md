# 不幸的是，您仍然需要阅读服务条款文档

> 原文：<https://lifehacker.com/you-still-need-to-read-terms-of-service-documents-unfo-1825388590>

有人阅读他们购买或注册的任何东西的服务条款吗？我的意思不是“看它一眼，承认它的存在；我的意思是坐下来喝杯咖啡(也许是一个玻璃瓶，这取决于文件有多长),浏览制造商或公司想让你知道的一切。如果你不这样做，我们不会责怪你，但你应该这样做。



你可能听说过 [网站的服务条款；最近没看](https://tosdr.org/) 。由于大科技最近面临的所有隐私问题，“平易近人的服务条款”的概念正在流传，最近来自 [【连线】](https://www.wired.com/story/terms-of-service-didnt-read/) 的一篇简介再次将聚光灯对准了这个第三方网站( [于 2012 年推出了](https://lifehacker.com/tos-dr-is-a-one-stop-shop-for-webapp-terms-of-service-r-5932853#_ga=2.242232642.1186395594.1523885983-542336946.1523479293) )。很高兴看到 ToS；博士还在线。然而，该行业借用 ToS 的一页了；博士的方法，并使*他们的*文件像这个网站一样容易阅读，因为 ToS；DR 不应该为他们做所有的事情，你也不应该信任它。

### 一个 TL；ToS 博士；速度三角形定位法(dead reckoning)

ToS；DR 采用了那些冗长的服务条款文档，并试图总结出它们最突出的要点，这样你就可以很快看到你要做的事情，而不必花一个下午的时间去解读法律术语。例如，考虑它在 GitHub 上的列表:

*   ***+** 你没有授予 github 任何版权许可*
*   ***——**变化随时可能发生，有时毫无察觉*
*   ***-** 你应该保护并赔偿 GitHub*
*   ***+** 您的个人信息仅用于有限目的*
*   ***-** 您的账户可以随时因任何原因被暂停，您的数据也可以被删除*

就是这样。你可以点击查看 GitHub 在 ToS 的列表；看到引用的例子，该网站用来创建这些小要点，但这里是我与 ToS 的症结所在；虽然该网站有助于了解软件、网站和服务如何运行其业务，以及这可能会对您产生何种影响，但试图找到最新的服务条款文档似乎是 ToS 的一场斗争；博士

### 跟踪 ToS 的更新是一项艰巨的工作

让我们看看在 ToS 上市的 GitHub 又是 DR——似乎自 2012 年以来就没有更新过。当你点击查看更多关于 ToS 的细节时；博士提供的要点和引文——可能来自 GitHub 的 ToS 的某个版本——在最新的文档中找不到。

例如，尝试搜索 ToS；博士引用了该网站第一条关于 GitHub 如何对待你上传到它上面的内容的话“我们不主张任何知识产权”。这句话在 GitHub 的 ToS 中已经不存在了。

这并不一定意味着 GitHub 改变了对待你的内容的方式；它只是更新了一点语言。这里有一个 ToS 的快速对比:博士的引用文本与 GitHub 的最新文本:

[**ToS；博士**](https://tosdr.org/#github)
*“我们对您提供给服务的材料不主张任何知识产权。您上传的个人资料和材料仍归您所有。但是，通过将[…]您的存储库设置为公开查看，您同意允许其他人查看和派生您的存储库。”*

[**GitHub**](https://help.github.com/articles/github-terms-of-service/#d-user-generated-content)
*“您保留对您创建或拥有的内容(“您的内容”)的所有权和责任。如果你张贴任何不是你自己创作的或不拥有权利的东西，你同意对你张贴的任何内容负责；您将只提交您有权发布的内容；并且您将完全遵守与您发布的内容相关的任何第三方许可。”*—D 部分，“用户生成的内容”，第三小节，“内容的所有权、发布权和许可授权。”

(GitHub 关于存储库的部分在关于授予其他用户的许可的单独章节中。)

### 你应该自己阅读服务条款

而 ToS；DR 抓住了 GitHub 正在谈论的要点，但摘要网站似乎在细节上有所欠缺。此外还有差异。举个例子，引用 GitHub 的声明，ToS；DR 标记为 bad:“您同意在注册服务时提供您的合法全名。这并不妨碍你使用假名。”

我翻遍了 ToS 的所有三份文件；DR 在其摘要中链接——GitHub 的 [服务条款](https://help.github.com/articles/github-terms-of-service/) 、 [隐私声明](https://help.github.com/articles/github-privacy-statement/) 和 [安全文档](https://help.github.com/articles/github-security/)——并没有发现对完整法律名称的要求。事实上，GitHub 的 ToS 现在说的正好相反:

> *"您必须提供有效的电子邮件地址才能完成注册过程。要求提供的任何其他信息(如您的真实姓名)都是可选的，除非您代表某个法律实体接受这些条款(在这种情况下，我们需要关于该法律实体的更多信息)，或者如果您选择了付费帐户，在这种情况下，额外信息对于计费是必要的。”*

这是件大事吗？不完全是，但算是吧。这是一个很好的提醒，总结有助于解读一个笨拙的 ToS，但你真的应该自己浏览文件，以确保你直接从源头获得关于公司政策的最准确、最新的信息。

更重要的是，你(和你的朋友)应该要求你关心的公司用日常语言重写他们蹩脚的、令人昏昏欲睡的服务条款。GitHub 采用了这种方法，提供了每个主要部分的有用的、易于理解的摘要。例如，以下是其七点“用户生成内容”部分的快速总结:

*" **简而言之:**您拥有您创建的内容，但您允许我们对其拥有一定的权利，以便我们可以显示和共享您发布的内容。您仍然对您的内容拥有控制权，并对其负有责任，您授予我们的权利仅限于我们提供服务所需的权利。如果需要，我们有权删除内容或关闭账户。”*

很可爱。

脸书——每个人最喜欢的隐私出气筒(和)在上个月 [剑桥分析](https://lifehacker.com/how-to-find-out-if-cambridge-analytica-had-your-data-1825155752) 的大爆炸后也采取了同样的处理方式。正如该公司在 4 月初的一篇博客中所写的:

> “用白纸黑字向人们展示我们的产品是如何工作的，这很重要——这是人们就自己的隐私做出明智决定的方式之一。因此，我们提议更新我们的服务条款，其中包括我们对每个使用脸书的人的承诺。我们用更易读的语言解释我们提供的服务。我们还在更新我们的数据政策，以更好地说明我们收集的数据以及我们如何在脸书、Instagram、Messenger 和其他产品中使用这些数据。”

至少可以说，脸书两份服务条款文件之间的差异是不和谐的。 [一个](https://www.facebook.com/legal/terms/update) ，你可能真的想读； [另一个](https://www.facebook.com/legal/terms/) ，你可能想在尝试之前准备好一个枕头。

完全没有理由为什么每家公司不效仿 ToS；博士的方法:简单明了的(更短的)服务条款，给你所有你需要知道的信息，而不会让你厌烦得要死。我们赞扬 ToS；但你不应该依赖第三方工具来让你了解你最喜欢的公司的条款和条件。现在是这些服务文档的来源收拾残局的时候了。
